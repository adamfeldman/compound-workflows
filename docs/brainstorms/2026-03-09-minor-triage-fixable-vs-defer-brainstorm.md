# MINOR Triage: Fixable vs Defer-Worthy

## What We're Building

Replace the binary MINOR triage (batch-accept or review individually) with a three-category pattern: LLM pre-sorts findings into "fixable now," "needs manual review," and "no action needed," proposes concrete fixes for fixable items, and the user confirms with partial acceptance. A subagent generates the categorization and fix proposals to keep the orchestrator context lean.

### Scope

- **brainstorm.md** Phase 3.5 Step 3 — MINOR batch triage
- **deepen-plan.md** Synthesis Gate Step 4 — MINOR batch triage
- **deepen-plan.md** Red Team Step 3 — MINOR batch triage
- **plan-consolidator.md** — batch triage pattern for guardrailed items
- **Fixability criteria** defined with concrete examples: low effort AND low risk AND obvious fix (all three must hold)
- **Three-category sorting** — fixable now / needs manual review / no action needed
- **Partial batch acceptance** — user can cherry-pick which fixes to apply
- **Post-fix verification** — re-read document after applying fixes to verify edits match proposals
- **Subagent delegation** — categorization and fix proposals generated by subagent, summary surfaced to orchestrator

### Out of Scope

- Changing the severity system (CRITICAL/SERIOUS/MINOR stays as-is)
- Auto-fixing without user confirmation (user always confirms)
- Changing how CRITICAL/SERIOUS items are triaged individually (they stay as "Valid/Disagree/Defer" — no fix proposals for higher severities)

## Why This Approach

### Problem

Current MINOR triage is binary: batch-accept (acknowledge all, no changes) or review individually (high interaction cost). Early evidence from completed brainstorms suggests many MINORs are trivially fixable — "add a sentence," "rename a term," "remove a contradiction." The user's stated preference is "fix all for MINOR" but the actual resolution pattern shows individual fixes applied. The current flow forces the user to either lose easy value (batch-accept) or pay disproportionate interaction rounds (review individually).

### Solution: Pre-sorted batch with proposed fixes

**Three categories:** The LLM sorts each MINOR into one of:

1. **Fixable now** — meets all three criteria (below), LLM proposes a concrete edit
2. **Needs manual review** — valid finding but too complex for a quick fix, presented individually
3. **No action needed** — observation with no concrete edit implied. Records whether the reason is "not an issue" or "actively disagree" for provenance.

**Fixability criteria** (all three must hold):
1. **Low effort** — a one-line or few-line edit, not a design change
   - Pass: "Add rationale for X exclusion → append one sentence to Decision 5"
   - Fail: "Restructure the precedence chain to address conflict handling"
2. **Low risk** — safe to change without ripple effects
   - Pass: "Rename 'cache' to 'context retention' in one section"
   - Fail: "Change a term used in 5+ other documents"
3. **Obvious fix** — no judgment calls, the edit is self-evident
   - Pass: "Add review.md to the Out of Scope list"
   - Fail: "Decide whether env vars should supplement or replace the config approach"

Pre-sorting reduces *interaction rounds* (one batch decision instead of N individual responses), not reading. The user still reads the proposed fixes before confirming — the efficiency gain is fewer back-and-forth cycles.

The user can partially accept — cherry-pick which fixable items to apply (e.g., "Apply 1, 3, skip 2") instead of all-or-nothing.

### Why pre-sorted batch, not tagged list

The pre-sorted approach groups fixable items together with their proposed edits, making the batch decision clearer. A tagged list intermixes categories, requiring the user to visually filter. Pre-sorting matches the observed workflow preference: "fix the easy ones, wave through the rest."

### Rejected alternatives

- **Confidence-gated auto-grouping** — LLM assigns confidence scores and auto-applies above threshold. Rejected: adds complexity without meaningful safety improvement over user confirmation. The three criteria already serve as the confidence gate.
- **Sample-first review** — Show 2-3 examples, user approves the pattern, LLM applies rest. Rejected: MINORs are heterogeneous, sampling doesn't represent the batch.
- **Dry-run diff approval** — Generate actual file diffs for user review. Rejected: too heavyweight for brainstorm-level document edits. Appropriate for code changes, not prose.
- **Apply top-K safe fixes** — Fix only the N most confident items. Rejected: arbitrary cutoff doesn't align with the fixability criteria.

## Key Decisions

### 1. LLM proposes, user confirms with partial acceptance

The LLM pre-categorizes MINORs and proposes fixes. The user can apply all, cherry-pick specific items, review individually, or acknowledge all. One decision point with flexible granularity.

**User rationale:** "LLM proposes batch, user confirms once" — but with partial acceptance so one wrong fix doesn't force full individual review.

### 2. Fixability requires all three criteria with concrete examples

A MINOR is "fixable now" only if it's low effort AND low risk AND obvious. Each criterion has pass/fail examples in the skill for calibration. Items that fail any criterion but are still valid go to "needs manual review" (individual triage), not "no action needed."

**User rationale:** "All three criteria must hold" with "add examples + accept judgment" — conservative bar with concrete calibration, but ultimately LLM judgment.

### 3. CRITICAL/SERIOUS triage unchanged

CRITICAL and SERIOUS items keep the existing "Valid/Disagree/Defer" individual triage without fix proposals. Rationale: CRITICALs often need structural shifts where a pre-generated fix anchors to superficial solutions. The scope stays focused on MINOR triage improvement.

**User rationale:** "Keep unchanged" — avoids anchoring risk and scope creep. Simpler.

### 4. Three categories, not two

"Acknowledge only" conflated "not an issue" with "valid but too hard to fix." Now: fixable (apply), needs manual review (individual triage), no action (acknowledged with provenance). Valid-but-complex MINORs never get silently waved away.

**User rationale:** "Combo of split into three categories and route complex to individual" — nothing valid gets dropped.

### 5. Subagent generates categorization

Categorization and fix proposals are generated by a subagent that writes to disk, consistent with context-lean convention. The orchestrator reads a summary and presents it to the user.

**User rationale:** "Subagent generates, summary surfaced" — context-lean compliance.

### 6. Post-fix verification

After applying batch fixes, re-read the document and verify each edit matches the proposal. Catches implementation drift. Consistent with plan-consolidator's existing re-verification pattern.

**User rationale:** "Add verification step" — even for MINORs, verify what was applied matches what was proposed.

### 7. Applies to all triage points

Consistent pattern across brainstorm.md (1 point), deepen-plan.md (2 points), and plan-consolidator.md (1 point). Four total triage points updated.

**User rationale:** "All triage points" — consistency everywhere.

### 8. Align fixability taxonomy with plan-consolidator during planning

The plan-consolidator uses "unambiguous vs judgment" while this uses "low effort + low risk + obvious." These are conceptually similar but defined differently. Should be aligned during planning to avoid two parallel fixability taxonomies in the same plugin.

## Integration Changes

### MINOR batch triage (3 command locations + 1 agent)

Replace:
```
"N MINOR findings remain. Review individually, or batch-accept as acknowledged?"
- Batch-accept
- Review individually
```

With (subagent writes full categorization to disk, orchestrator presents summary):
```
"N MINOR findings from red team:

**Fixable now** (low effort, low risk, obvious fix):
1. M2: [summary] → [proposed edit]
2. M4: [summary] → [proposed edit]
3. M7: [summary] → [proposed edit]

**Needs manual review** (valid but complex):
4. M5: [summary] → [why it needs manual review]

**No action needed** (observations):
5. M1: [summary] → [not an issue / actively disagree]
6. M3: [summary] → [not an issue / actively disagree]

Apply fixable batch and acknowledge the rest?"

- Apply all fixes + acknowledge no-action (Recommended)
- Apply specific fixes (e.g., "1, 3") + acknowledge rest
- Review individually
- Acknowledge all (no fixes)
```

Items in "Needs manual review" are then presented individually with standard triage options.

### Post-fix verification

After applying fixes:
1. Re-read the modified sections of the document
2. Verify each edit matches the proposed fix
3. If drift detected, flag to the user before proceeding

## Resolved Questions

### 1. Should CRITICAL/SERIOUS triage also get fix proposals?

No — dropped during red team triage. CRITICALs often need structural shifts where pre-generated fixes anchor to superficial solutions. Keeps scope focused on MINOR triage. (Red team finding: Gemini flagged anchoring risk, Opus flagged scope contradiction.)

### 2. Should the categorization happen inline or via subagent?

Subagent — generates categorization and fix proposals, writes to disk. Orchestrator reads summary. Consistent with context-lean convention. (Red team finding: Opus flagged context cost of inline presentation.)

### 3. How thin is the empirical basis?

Early evidence from 3 brainstorms (19 MINOR findings total, 0-71% fixable range) suggests the pattern has value but the specific percentages are soft. The design doesn't depend on a specific fixable ratio — it works whether 30% or 70% of MINORs are fixable. (Red team finding: OpenAI and Opus flagged thin evidence. Claim softened from "50-70%" to "many MINORs.")
